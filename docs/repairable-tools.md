# 可修复工具 — Repairable Tools（构想，待评审）

> 状态：**构想记录**，尚未排期。待技能系统 v2 与沙箱方案思考成熟后一起评审。
> 关联：`docs/voyage-loop.md`（任务循环）、`docs/skill-system.md`（技能系统 v1）、
> `docs/anthropic-agent-design.md`（调研依据）。
>
> 起因（2026-07-16 实际事故）：`wiki.fetch_extract` 因 PDF 文本层带 `0x00` 触发
> Postgres `CharacterNotInRepertoireError`，确定性 bug 导致建库任务反复失败；
> LLM 重规划产生"假修复"（新步骤标题承诺清洗，但动作代码不可变），断点重试无效，
> 闭环断在「等人修代码」。本文回答：如果工具本身可以被修复，这个闭环能否在平台内闭合，
> 以及这个思路的泛化边界在哪。

## 1. 核心想法

把一部分工具（action 依赖的确定性代码）从「平台发版才能改的冻结代码」升级为
「平台内可版本化、可测试、可修复的一等 artifact」。失败闭环从：

```text
确定性 bug → 任务反复失败 → 等人修代码、重启 worker → 手动断点重试
```

变为：

```text
确定性 bug → 相同错误签名判定（voyage-loop §5.1）→ 原任务 paused_error
           → 自动开 tool_repair 任务：复现 → 失败输入固化为回归测试 → 修码
           → 全量回归通过 → 人工闸门审批 → 发布新版本 → 原任务断点续跑
```

这不是全新的信任假设：平台已在受控环境执行 LLM 生成的代码
（`experiment.setup` 实验代码、`experiment.figures` 绘图脚本）。
可修复工具是把这个既有信任从实验沙箱推广到平台工具层，并配上更强的 harness。

## 2. 泛化边界：按副作用画线

平台"确定性 vs 判断性分离"铁律升级为三分：
**判断性（LLM）/ 可修复确定性（边缘适配器）/ 冻结确定性（核心）**。

| 层 | 例子 | 可修复？ |
| --- | --- | --- |
| 纯变换 / 适配器 | PDF 抽取、metric 行解析、站点 scraper 适配、格式转换、绘图脚本 | ✅ 理想对象：无副作用、可沙箱、失败可复现、测试可判定 |
| 提示词 / 描述 / rubric | 技能系统 v1 已覆盖（guidance/rubric/persona） | ✅ 已证明可行（[MultiAgent]：agent 重写工具描述使任务耗时降 40%） |
| 有副作用但幂等可测 | 带水位线的抓取、upsert 入库 | ⚠️ 谨慎：需事务隔离 + 集成测试，后期再评估 |
| 核心层 | Voyage engine、DB models、security、闸门逻辑 | ❌ 永远冻结，只走人类发版 |

泛化性成立的根本原因：**运维失败绝大多数住在适配器层**（PDF 千奇百怪、网站改版、
API 换字段——现实世界的脏东西全在边缘），而核心层 bug 频率低、后果重，本就该走
正式发版。[BEA] 论证 coding 是 agent 理想场景正因"solutions are verifiable through
automated tests"——修纯函数是代码任务里验证性最强的一类。

## 3. 四道 harness（缺一不可，否则是自毁装置）

1. **失败输入即回归测试**：修复验收 = 老 golden 用例全过 + 本次失败输入固化为新
   测试用例并通过。这是人类工程师工作流的机制化（[Harness] 方法论）。
2. **防 reward hacking 的产出质量验收**：修复 agent 最省事的"修法"是吞异常/返回
   空文本——不报错但工具废了。验收必须含输出质量检查（抽取文本长度下限、关键章节
   存在等），直接复用 voyage-loop §6 检查注册表（`min_count`/`schema_valid`）。
   同源教训：[Harness]"禁止删测试让测试通过"。
3. **版本化 + 快照**：工具像技能一样带版本；voyage 启动时把工具版本快照进
   checkpoint——机制复用技能系统的 `_ensure_skills_snapshot`，事后可回放
   「这次任务用的是工具哪个版本」。本质上这是**技能系统 v2：从声明式数据升级到
   沙箱代码**（skill-system.md §0 v1 明确禁了任意代码执行；此处是自觉破例，
   对价 = 沙箱 + 测试 + 版本化 + 闸门）。
4. **修复过闸门**：工具变更初期一律人工审批（与远程写同级），闸门 payload 带
   diff + 失败用例 + 回归结果；跑稳后可对「纯变换 + 全量测试通过」的修复放宽为
   自动合入、事后通知。

## 4. 与任务系统的合流

- `tool_repair` 本身是一个 voyage kind，且按 voyage-loop 分档属于**档 0/1**：
  计划固定（复现 → 固化测试 → 修码 → 回归 → 闸门 → 发布），验收全是 rules-based——
  不需要动态循环。
- 触发点接在 voyage-loop 的「相同错误签名判定」上：连续相同错误 → 判定确定性失败 →
  原 voyage 转 `paused_error` + 自动创建 tool_repair voyage → 修复发布后原任务断点续跑。
- 三条设计线在此闭合成完整自愈环：失败签名判定（何时修）、可修复工具（修什么）、
  断点恢复（修完怎么继续）。

## 5. 代价与落地顺序

代价：沙箱工程（受限 import、资源/时间限制）是真投入；调试多一个"工具版本"维度
（快照机制兜底）；golden 测试资产需要积累（如 golden PDF 集）。

落地顺序（评审通过后）：

1. 试点：挑 2–3 个最常坏的适配器（PDF 抽取、metric 解析）抽成可修复工具，
   定义 input/output contract + golden 用例集；
2. 修复全程人工闸门，跑通「事故 → 自动修复提案 → 审批 → 续跑」闭环；
3. 稳定后评估自动合入的放宽条件与扩面（scraper 适配等）；
4. 不做的事：不把所有 action 变成可编辑；核心层永远不进这套机制。

## 6. 待评审的开放问题

- 沙箱技术选型：subprocess + 受限环境 / WASM / 独立容器？与技能系统 v2 共用一套？
- 工具 contract 的表达：JSON Schema 够不够？纯函数签名如何声明与校验？
- 修复 agent 用平台内 LLM 循环实现，还是外挂 coding agent（如 Claude Agent SDK）？
- 多项目共享工具的版本策略：项目级 pin 还是全局最新？与技能市场的分发机制是否统一？
- golden 用例的存储与脱敏（PDF 可能有版权/隐私问题）。
